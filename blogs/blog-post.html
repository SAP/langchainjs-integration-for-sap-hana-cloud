Vector Search with SAP HANA Cloud and LangChain.js

Imagine your customer types “something comfortable for working from home” into your shop’s search bar. There’s no “comfortable” column in your product table. No “work from home” tag either. And yet your store somehow needs to surface the ergonomic chair, the noise-cancelling headphones, and the standing desk.

That gap between what people say and what your database stores is exactly where vector search lives. With @sap/hana-langchain, your SAP HANA Cloud database doesn’t just store your data. It understands it.

The full walkthrough, with every line of code, every output, and every config option, lives on GitHub: blogs/blog-post.md. The sections below focus on the ideas worth remembering, paired with the bits of code and output that bring them to life.

What it is

@sap/hana-langchain is the official open-source bridge between SAP HANA Cloud’s Vector Engine and LangChain.js. It turns your enterprise database into a first-class AI retrieval layer for RAG, semantic search, recommendations, and intelligent Q&A, all running where your data already lives.

Two lines and you have a vector store:

npm install @langchain/core @langchain/classic langchain
npm install @sap/hana-langchain

Searching by intent, not by keywords

Let’s walk through the real story. A customer is setting up their home office. They open your store and search:

“comfortable work from home setup”

Your catalogue has hundreds of products. Keyword search? Zero relevant hits. With HanaDB? Watch what happens.

import { HanaDB, HanaInternalEmbeddings } from "@sap/hana-langchain";

// Embeddings happen INSIDE HANA. No external API. No data leaving your DB.
const embeddings = new HanaInternalEmbeddings({
  internalEmbeddingModelId: "SAP_NEB.20240715",
});

const store = new HanaDB(embeddings, {
  connection: client,
  tableName: "PRODUCT_CATALOG",
});
await store.initialize();

const results = await store.similaritySearch(
  "comfortable work from home setup",
  2,
);

The output:

[
  Document {
    pageContent: 'Ergonomic office chair with lumbar support and adjustable armrests',
    metadata: { category: 'furniture', price: 449, in_stock: true }
  },
  Document {
    pageContent: 'Professional espresso machine with built-in grinder',
    metadata: { category: 'appliances', price: 899.99, in_stock: false }
  }
]

The chair makes obvious sense. The espresso machine? Also surprisingly on-brand for “working from home.” The model picked up on intent that no SQL LIKE ever could.

Cosine vs. Euclidean: pick your distance

So how does similaritySearch actually decide which products are “closest” to the query? Each document and the query are turned into vectors, and then HanaDB applies a distance metric to measure how far apart they are. The library supports two: cosine (the default) and Euclidean. You pick which one you want with a single config option when you create the store.

The choice matters because the two metrics score the same set of candidates differently. Take the query “high quality audio headphones” and run it under both strategies. Same data, same embedding model, different scoring philosophy:

COSINE Similarity Results (higher = more similar)
[0.7219] Wireless noise-canceling headphones with premium sound quality
[0.5506] Professional studio microphone for podcasting and streaming
[0.5206] Bluetooth speaker with deep bass and 20-hour battery

EUCLIDEAN Distance Results (lower = more similar)
[0.7086] Wireless noise-canceling headphones with premium sound quality
[0.9048] Professional studio microphone for podcasting and streaming
[0.9251] Bluetooth speaker with deep bass and 20-hour battery

Both metrics pick the same winner here, but notice the gap between first and second place: cosine spreads the candidates by relative similarity, Euclidean by raw distance in vector space. As a rule of thumb, cosine is the right default for text, because it only cares about the direction the vectors point in, not how long they are. Euclidean is a better fit when the actual position of the vectors carries meaning, for example when your embeddings come from numerical features rather than text. If you’re not sure which to pick, leave it on cosine.

Adding diversity when you need it: MMR

Sometimes “most similar” is exactly what the user wants. A shopper who searches for “wireless headphones” expects a list of wireless headphones, and similarity search delivers that perfectly well.

But there are spots in the same product journey where pure similarity hurts the experience. The classic case is the “customers also viewed” strip on a product detail page. A shopper looking at a premium open-back studio headphone doesn’t need to see four more premium over-ear headphones below it. They’d be much better served by a strip that also shows them a different form factor, a different price tier, or a noise-cancelling alternative.

That’s the job of Maximal Marginal Relevance (MMR). Instead of returning the top-k most similar documents, MMR picks results that are both relevant and different from each other. You retrieve a larger candidate pool; MMR walks it, picking the next item that maximises relevance while penalising redundancy with what’s already been chosen. One lambda knob tunes the balance between relevance and diversity.

Same vector store, same query, two different methods. First, plain similarity search:

// Plain similarity search: top-4 candidates ordered by closeness alone
const similar = await vectorStore.similaritySearch(
  "premium wireless over-ear headphones",
  4,
);

similaritySearch top 4:
  1. AudioMax Pro 5     (over-ear-wireless, $349.99)
  2. AuroraSound Pro    (over-ear-wireless, $379.99)
  3. AudioMax Buds 5    (true-wireless,     $299.99)
  4. QuietShield Ultra  (over-ear-wireless, $429.99)

Three of the four results are premium over-ear wireless headphones, almost interchangeable for the shopper. Not a useful “customers also viewed” strip.

Now the same query, through MMR:

// MMR: pick 4 from a wider candidate pool while penalising redundancy.
// lambda = 0.5 balances relevance and diversity evenly.
const recommendations = await vectorStore.maxMarginalRelevanceSearch(
  "premium wireless over-ear headphones",
  { k: 4, fetchK: 12, lambda: 0.5 },
);

maxMarginalRelevanceSearch top 4:
  1. AudioMax Pro 5     (over-ear-wireless, $349.99)
  2. FlexFit Sport      (neckband,           $89.99)
  3. AuroraSound Pro    (over-ear-wireless, $379.99)
  4. TuneCore 510       (on-ear-wireless,    $49.99)

The top match is still the most relevant premium over-ear pair, but the next three deliberately span different form factors and price tiers. The shopper now sees an actual mix to compare, all from one method swap on the same vector store and the same product catalogue.

Use similaritySearch when the goal is the closest match. Reach for maxMarginalRelevanceSearch when the goal is coverage: recommendation strips, RAG context that spans multiple sub-topics, related-articles modules, or anywhere a single dominant theme would crowd out everything else.

Filtering that speaks JavaScript

Semantic search is great. Semantic search plus structured filters is unbeatable. HanaDB’s filter syntax will feel instantly familiar to anyone who’s ever written a MongoDB query:

// In-stock electronics under $500
const filter = {
  $and: [
    { in_stock: true },
    { category: "electronics" },
    { price: { $lt: 500 } },
  ],
};

const results = await store.similaritySearch("best audio quality", 10, filter);

And the result is exactly what the filter promised: only in-stock electronics, all under $500, ranked by audio quality:

[
  { pageContent: 'Premium wireless headphones with spatial audio...',  price: 349.99, in_stock: true },
  { pageContent: 'Budget wireless earbuds with decent sound...',       price:  49.99, in_stock: true },
  { pageContent: 'Bluetooth over-ear headphones with active noise...', price: 249.99, in_stock: true },
  { pageContent: 'Wireless noise-canceling headphones with 30-hour...',price: 299.99, in_stock: true }
]

You get the full operator suite ($lt, $lte, $gte, $between, $in, $nin, $like, $contains, $and, $or) all compiled down to native HANA SQL. No JavaScript-side filtering. No surprises at scale.

Letting the LLM write its own filters

Here’s where it gets fun. What if a user types “Show me AudioMax electronics under $400 with good ratings”? You could build a custom intent parser. Or you could let the LLM do it for free, with the HanaTranslator:

const retriever = SelfQueryRetriever.fromLLM({
  llm,
  vectorStore: store,
  documentContents: "Product description for an e-commerce catalog",
  attributeInfo,                              // describe your metadata
  structuredQueryTranslator: new HanaTranslator(),
});

await retriever.invoke("Show me AudioMax electronics under $400 with good ratings");

Behind the scenes, the LLM emits:

{
  $and: [
    { brand: "AudioMax" },
    { category: "electronics" },
    { price:  { $lt:  400 } },
    { rating: { $gte: 4 } }
  ]
}

Natural language in. Structured filter out. Vector search on top. Zero glue code.

There’s more in the box

What you’ve seen so far is the foundation. The library ships with three more capabilities, each big enough to deserve its own deep-dive:

  • Cross-encoding reranking with HanaReranker. Re-score your top candidates for surgical-precision results, the secret sauce behind great RAG and customer-facing search.
  • Knowledge graphs & SPARQL with HanaRdfGraph. Query RDF data, build natural-language Q&A over relationships, and combine graph reasoning with vector search.
  • Performance at scale with Map Merge, HNSW indexes, and specific metadata columns. Roughly 9× faster bulk inserts, sub-second search at million-document scale, and faster filtered queries.

Stay tuned for more updates and deep dives into each of these capabilities!
